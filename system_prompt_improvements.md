# System Prompt 改进建议

基于 Agent 实际运行中的踩坑体验，以下是 system prompt 中需要优化的描述及建议。

---

## 1. "禁止直接输出"——增加后果说明

**当前措辞**（团队协作与群聊规则 第3条）：
> - **禁止直接输出**：不要直接输出文字回复，必须通过工具调用。

**问题**：模型知道"不能"，但不知道"为什么不能"，缺乏动力严格遵守。

**建议改为**：
> - **禁止直接输出**：不要直接输出文字回复，必须通过工具 send_chat_msg 发送到房间。操作者和其他成员只会收到发送至房间的消息，直接输出的文字他们看不到。

---

## 2. send_chat_msg 返回值措辞

**当前返回值**：
> "你当前的发言任务已视为完成。如果没有其他辅助工具（如查看信息）需要调用，请立即执行 finish_chat_turn 提交并结束本轮。"

**问题**："发言任务已视为完成"暗示整个任务结束了，容易让模型跳过 still-pending 的工具调用。

**建议改为**：
> "消息已送达房间。如果你还有其他工具需要调用，请继续；如果本轮操作已全部完成，请调用 finish_chat_turn 结束本轮。"

---

## 3. 初次入场规则——明确轮次定义

**当前措辞**（团队协作与群聊规则 第7条）：
> - 当房间刚创建时，简单问候和自我介绍即可，无需过度闲聊。
> - 经过约两轮交流后，如果没有明确任务，请停止发言并调用 `finish_chat_turn()` 跳过后续轮次。

**问题**："约两轮交流"定义模糊。是指每个成员各发言一次算一轮？还是总共两轮调度循环？

**建议改为**：
> - 当房间刚创建时，简单的自我介绍和问候即可，无需过度闲聊。
> - 当大家已经轮流完成自我介绍，并且你发言两次之后，如果没有明确任务，请停止发言并调用 `finish_chat_turn()` 跳过后续轮次。



---

## 5. 待命约束——避免模糊措辞

**当前措辞**（团队协作与群聊规则 第5条）：
> - 待命状态下直接说明当前无任务、处于待命即可，不要用"正在梳理""准备推进"等措辞填充。

**问题**：这条写得很明确，已经不错。但结合第7条"停止发言并调用 finish_chat_turn()"，模型可能会困惑：是发一条"当前无任务，待命中"再 finish，还是直接 finish？

**建议补充**：
> - 如果上一轮已经说明过待命状态且无新任务下达，可直接调用 `finish_chat_turn()` 跳过，无需重复说明。

---

## 优先级

| 优先级 | 条目 | 影响 |
|--------|------|------|
| 🔴 高 | 1. 禁止直接输出后果说明 | 直接导致操作者看不到消息 |
| 🟡 中 | 2. send_chat_msg 返回值 | 可能跳过后续工具调用 |
| 🟡 中 | 3. 初次入场轮次定义 | 造成不必要的多轮空转 |
| 🟢 低 | 5. 语言适配 | 边界场景 |
| 🟢 低 | 6. 待命补充 | 小优化 |
